# Session Protocol

This is the startup map for the repo's agent session system. Keep it in
context; load lifecycle skills and their references only when the user asks for
that lifecycle action.

## Command Routing

Sessions are opt-in. Default to no session and do not create journal state
unless the user explicitly asks.

| User intent | Action |
|-------------|--------|
| "session setup", "setup sessions", "onboard me" | Invoke `session-setup`. |
| "new session", "start a session" | Invoke `session-new`. |
| "continue session N" | Invoke `session-continue` with `N`. |
| "session close", "wrap up", "end the session" | Invoke `session-close`. |
| Anything else | Help normally; do not journal. |

If the request sounds session-related but the mode is ambiguous, ask before
priming anything.

## Journal Root

The journal root is the checkout whose `.journal/` directory should be read and
written.

Resolve it in this order:

1. If the current branch is `journal/<github-login>`, use the current checkout.
2. If `gh api user --jq .login` succeeds and `wt list --format=json` shows a
   worktree for `journal/<github-login>`, use that worktree.
3. If the current checkout has a local `.journal/`, use it as a legacy/local
   fallback.
4. If no journal root exists, continue without journal context unless the user
   asked for setup or a session lifecycle command.

`session-setup` is the only lifecycle skill that creates or opens the personal
journal branch. `session-new`, `session-continue`, and `session-close` must stop
and tell the developer to run `session-setup` if no personal journal worktree
exists.

## Startup Context

Before substantive repo work:

1. Read this file.
2. Resolve the journal root, if any.
3. If `<journal-root>/.journal/SKILLS.md` exists, load every required skill it
   lists.
4. If `<journal-root>/.journal/TECH_NOTES.md` exists, read it.
5. Read recent session summaries only when they are likely to matter: session
   lifecycle commands, substantial code/config/docs changes, architecture work,
   multi-step research, or tasks that mention prior work. Use the latest three
   closed sessions with `SUMMARY.md`; skip sessions without summaries.

Use `<journal-root>/.journal/INDEX.md` as the full session map. Search it for
task-relevant history and read only the matching `SUMMARY.md` files.

Do not read session `NOTES.md` files during ordinary startup. Read a session's
`NOTES.md` only when continuing that session, when a lifecycle skill instructs
you to, or when the user explicitly asks for it.

## Development Stack Startup

YACD implementation sessions have a required local environment startup step.
This is session-level setup, not a per-turn or per-task hook. Do not treat an
implementation session as ready for implementation work until this one-time
procedure is handled or the human explicitly waives it.

After the journal is primed and the implementation Worktrunk worktree has been
selected or created:

1. Run the startup command from the implementation worktree, not from the
   journal worktree:

   ```sh
   moon run root:dev-up
   ```

2. Confirm the command exits successfully. It starts Tilt in the background,
   waits for the controller to become ready, then returns.
3. If startup fails, stop and surface the failure before making unrelated code
   changes. Fixing the dev stack itself is the only exception.
4. Record a concise `NOTES.md` checkpoint when startup materially affects the
   session, especially after failures, repairs, or smoke-test results.

The stack is intentionally singleton per checkout family. Runtime ownership and
logs live under `.run/yacd-dev/` in the primary checkout, while commands should
still be run from the active implementation worktree. If the stack is already
running for the active session, do not restart it just because the user gives a
new task or the agent resumes after review.

## Skills

Always load and follow:

- `git`
- `worktrunk`

They should be installed repo-locally under `.agents/skills`. If either is
missing, tell the user the protocol install is incomplete before doing version
control work.

Before touching a technology or workflow, survey available skills and load the
task-relevant ones. Project-required skills belong in
`<journal-root>/.journal/SKILLS.md`; keep that file as a short list, not a
manual.

## Branch And Journal Invariants

- Personal journal branches are named `journal/<github-login>`.
- Journal branches are never PR'd or merged into the default branch.
- The default branch keeps `.journal/` ignored and untracked.
- Only a `journal/<github-login>` branch may use `git add -f .journal`.
- Implementation worktrees are created from the fetched default branch, never
  from a journal branch.
- Before closing implementation work, fail if any non-journal branch has tracked
  journal files: `git ls-files .journal` must print nothing.
- Lifecycle journal writes happen in the journal root, not in implementation
  worktrees.
- Before writing session state, lifecycle skills require a clean journal
  worktree, pull/rebase it, commit the mutation, and push the journal branch.
  On rejected push, fetch/rebase once and retry once; if conflicts remain,
  stop and surface them.
- Worktrunk worktrees must live under `.wt/` via
  `worktree-path = "{{ repo_path }}/.wt/{{ branch | sanitize }}"`. If config
  would place worktrees elsewhere, stop and tell the user.
- Integration work uses GitHub PRs and squash merges. Do not use `wt merge`,
  `wt step push`, or local `git merge` as the default integration path.

## Journal Structure

```
.journal/
├── INDEX.md
├── SKILLS.md
├── TECH_NOTES.md
└── <ID>/
    ├── NOTES.md
    └── SUMMARY.md
```

- Session IDs are zero-padded three-digit folders (`001`, `002`, ...).
- `INDEX.md` is the table of contents across sessions.
- `SKILLS.md` lists project-required skills.
- `TECH_NOTES.md` holds compact durable technical context and is editable.
- `NOTES.md` is the append-only running log for resuming an active session.
- `SUMMARY.md` is the curated postmortem written at closeout.

Exact templates and write procedures live in the lifecycle skills, especially
`session-new` for `NOTES.md` and `session-close` for `SUMMARY.md` / `INDEX.md`.

## During A Session

- Keep `NOTES.md` updated at meaningful checkpoints.
- Timestamp note headings in the user's local time as `YYYY-MM-DD HH:MM`.
- Correct earlier notes by appending a new entry, not by rewriting history.
- Update `TECH_NOTES.md` only for durable project context future agents need.
- Do not create or edit a separate `.journal/` inside an implementation
  worktree.

## Development Stack Shutdown

At explicit session close or end-of-session, agents must stop the local dev
stack unless the human explicitly asks to leave it running. Do not stop it at
ordinary turn boundaries, after each implementation task, or merely because the
agent is pausing for user review; the stack is intentionally kept warm across an
active session.

Run the shutdown command from the implementation worktree that owns the stack
when possible:

```sh
moon run root:dev-down
```

This stops the recorded Tilt process, removes Tilt-managed resources, deletes
the Kind cluster and local registry, and removes `.run/yacd-dev/` runtime
state. If shutdown fails, surface the failure clearly and do not claim the
session or environment is clean.

`session-close` still owns PR, merge, Worktrunk cleanup, and journal closeout.
The dev stack must be down before final closeout is reported.
