## Goal
Bring `internal/controller/cardanodbsync` up to the cardanonetwork controller bar (set in session 021 / PRs #38 and #39) in a single PR: typed condition vocabulary, file-by-file split mirroring the cardanonetwork peer, godoc bar pass, and the deferred-from-018 storage-default bleed (`LedgerBackend: "lsm"`, `NearTipEpoch: 580`) folded into `defaults.go`.

## Outcome
The goal was met. PR #40 merged as `b131069` with the squash title `refactor(cardanodbsync): split controller package, tighten godoc bar, type condition vocabulary`. CI and the review agent passed before merge. The review agent caught one P3 doc regression in `postgres_identity.go` (corrected in `911d1e3` before merge — see Lessons). Local `master` was fast-forwarded, the `refactor/controller-cardanodbsync` worktree and branch were removed, and the dev stack was stopped.

The 928-line `workload_builder.go` is gone. The controller package now matches the cardanonetwork file taxonomy (`doc.go`, `controller.go`, `apply.go`, `callbacks.go`, `builder.go`, `containers.go`, `resources.go`, `settings.go`, `defaults.go`, `validate.go`-equivalent absorbed into `builder.go`, `names.go`, `labels.go`, `annotations.go`, `conditions.go`, `errors.go`, `status.go`, `readiness.go`, `runtime_probe.go`) plus the cardanodbsync-domain-specific files (`database.go` for dependency resolution, `managed_postgres.go` for the managed Postgres builder entry point, `postgres_identity.go` for pure identity types). Largest non-test file is now 502 LOC (`resources.go`), versus the prior 928.

## Key Decisions
- **Single PR, not split.** Despite the size of the diff (25 files changed, +2798/-1832), one PR was the right call because every commit is behavior-preserving and the typed condition vocabulary cascades through every file. Splitting would have produced a sequence of partial states that each had to compile against the typed boundary.
- **Eight rejected architectural changes locked at plan time.** The user explicitly excluded: `KubernetesClient` port, `ReadinessProber` port, splitting the unified `runtimeProber` into Postgres+Ogmios, subpackage extraction for the runtime prober adapter, and mockery introduction. These would all diverge from the cardanonetwork peer and turn a refactor pass into separable cross-cutting decisions. They stand as deferred-to-separate-decision items.
- **Terminology canon: "dbsync workload" and "managed Postgres".** Confirmed at plan time. The two-container follower-node + db-sync Deployment is the "dbsync workload"; the owned DB is "managed Postgres" (tech-specific, not "managed database"); its Deployment is the "managed Postgres workload". This shows up consistently in godocs and identifier names.
- **Identity acceptance functions stay in `apply.go`, not callbacks.go.** The cardanodbsync identity-acceptance functions (`validateAcceptedDBSyncDatabaseIdentity`, `validateAcceptedManagedPostgresIdentity`, etc.) are reconciler methods that read live state from the PVC/Deployment and stamp accepted identity into in-memory status. They are not pure ApplyOwnedObject callbacks, so they belong in the apply orchestration layer alongside `handleDBSyncWorkloadApplyError`, not in `callbacks.go`.
- **In-container layout constants stay in `builder.go`, not `containers.go` or `defaults.go`.** The volume names, mount points, and file-name constants are consumed by all three of `builder.go`, `containers.go`, and `resources.go` (the builder embeds them in plan args; containers.go mounts them; resources.go declares the Volumes). Centralizing in `builder.go` matches the cardanonetwork pattern where `containers.go` owns its container-name constants but shared layout lives near the orchestrator.
- **`readyCondition` lost its `ConditionStatus` parameter.** Every caller passed `metav1.ConditionFalse` (aggregate Ready=True is constructed through `workloadsReadyCondition`'s `ctrlstatus.AggregateReady`). Caught by `unparam` lint; simplification was a one-line clean-up at the godoc-pass commit.

## Lessons
- **A wrong godoc in a security-sensitive area is a real regression, not a "doc nit."** The review agent caught one P3 in `postgres_identity.go` — the field comment on `managedPostgresAuthIdentityInput.Version` claimed the user-provided path uses Secret `ResourceVersion`, when the code actually hashes password material. The invariant ("identity is based on password material, not Secret resourceVersion metadata") is documented in TECH_NOTES.md, but my godoc would have steered a future maintainer the other way. Fix shipped in `911d1e3` before merge. Lesson for future godoc passes: when documenting code that exists to preserve a security invariant, treat the godoc itself as security-load-bearing — re-read the implementation while writing the comment, do not summarize from memory.
- **Plan ahead for review-agent doc audits when the package owns identity hashing.** The review agent's strongest catches were on cross-references between the code and the surrounding documentation; structural review (callbacks split, file boundaries, test mutations) was clean. Future refactor passes over identity / fingerprint / Secret code should add a discrete "re-read the invariants in TECH_NOTES.md, then re-read my new godocs against the implementation" step before opening the PR.
