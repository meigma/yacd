## 2026-05-20 12:50 — Kickoff
Goal for the session: draft a minimally viable root `DESIGN.md` for YACD that captures the initial vision and architecture strongly enough to bootstrap the first real prototype without trying to resolve every unknown.
Current state of the world: the repo is a fresh downstream copy of `../template-k8s`, still carrying the template `NginxDeployment` API and chart names. The product direction is a Kubernetes operator for Cardano developer environments: local-first, Kind/Tilt-friendly, namespace-scoped where practical, and aimed at builders rather than validators or stake pool operators. The operator should eventually stand up Cardano networks plus useful developer services such as Ogmios, Blockfrost-like APIs, or related tooling.
Plan: start with research before writing design text. First investigate `bloxbean/yaci` and adjacent Yaci tooling to determine whether it can be reused, partially reused, or treated mainly as influence for YACD's first prototype. Preserve findings in these notes as the investigation progresses.

## 2026-05-20 12:53 — Yaci repo split
Research checkpoint: `bloxbean/yaci` itself is not a Kubernetes-native environment manager; it is a Java Cardano mini-protocol library. The closest adjacent projects are `bloxbean/yaci-devkit` for local devnet orchestration and `bloxbean/yaci-store` for indexing/datastore/API behavior. Current investigation is using fresh GitHub metadata plus local shallow clones under `/tmp/yacd-yaci-research/`.
Current read: YACD probably should not treat `yaci` alone as the comparable product. The meaningful comparison is the Yaci stack: DevKit + Store + optional Ogmios/Kupo + core Yaci protocol library.

## 2026-05-20 12:56 — Yaci initial assessment
Research checkpoint: Yaci core is a real bespoke Java mini-protocol implementation, not just a wrapper over Ogmios or cardano-cli. Yaci DevKit is the closest product analogue to YACD, but it is process/home-directory/Docker-Compose oriented and delegates much of the runtime to real Cardano binaries plus optional Ogmios, Kupo, Yaci Store, and Yano. Yaci Store is a plausible optional indexer/API component, with useful Blockfrost-compatible ambitions, but its Java/Spring footprint and active compatibility/correctness issue surface make it a better optional profile than a required MVP dependency.
Current recommendation: let the Yaci stack strongly influence the first YACD prototype, but do not embed Yaci core in the Go operator or reuse DevKit as the operator control plane. Prototype a Kubernetes-native CRD that owns Cardano node resources and optional developer services; treat DevKit and Store as references or workload components to test, not as the architecture's center of gravity.

## 2026-05-20 13:04 — DevKit developer-experience takeaways
Yaci DevKit's strongest UX contribution is not any single component; it is the complete local developer loop. It gives developers a fast devnet bootstrap, funded default wallets, faucet/topup commands, UTxO/tip/info inspection, reset semantics, optional Blockfrost/Ogmios/Kupo-compatible endpoints, a browser viewer, SDK integration examples, CI-friendly NPM startup, rollback simulation, and MCP tools for agent-assisted testing.
YACD should emulate these affordances while translating the control plane into Kubernetes concepts. The first YACD design should probably frame the CRD around a developer environment, not just a node, and should expose status/connection details, generated test accounts or funding hooks, reset/recreate operations, and optional service profiles as first-class developer-facing features.

## 2026-05-20 13:12 — Two-component YACD shape
Emerging architecture discussion: YACD likely wants at least two user-facing components. The Kubernetes operator should own declarative environment state: node workloads, genesis/config material, optional service profiles, generated wallet secrets, baseline funding policy, status, and readiness. A companion CLI should own imperative developer actions: discover environments, print connection info, port-forward, top up an address, inspect wallets/UTxOs, reset a local devnet, tail logs, or open the viewer.
This split keeps Kubernetes-native reconciliation for long-lived desired state while avoiding awkward CRD modeling for one-off developer actions such as "top this wallet up right now." Lightweight custom in-cluster services may still be useful for faucet/topup or admin actions, with the CLI acting as a friendly client rather than putting all imperative behavior into the operator itself.

## 2026-05-20 13:18 — API shape direction
Emerging API direction: use multiple Kubernetes CRDs rather than one monolithic environment CRD. A primary environment/network CRD should express the Cardano node/devnet substrate, while supporting service CRDs should model distinct services such as Ogmios, indexers, viewers, or faucet/admin helpers. Each controller can reconcile its own workload and status, with references back to the primary environment.
To avoid making local standup verbose, add a developer-facing local config consumed by the companion CLI. The CLI can compile that single-pane config into the underlying Kubernetes CRDs and apply them to the target cluster. This preserves a simple checked-in developer config while keeping the cluster API decomposed and controller-friendly.

## 2026-05-20 13:24 — DevKit wallet/topup implementation check
Code check: Yaci DevKit uses a bespoke CLI/admin layer for wallet and faucet-style developer operations. In normal Haskell-node mode, topup and UTxO inspection do not go through Ogmios; they use Yaci local mini-protocol clients through the node socket. `AccountService` delegates to `LocalNodeService`, which loads genesis UTXO keys, queries UTxOs with local state query, builds transactions with cardano-client-lib, and submits with local tx submission. Default wallet listing is generated from a fixed test mnemonic via `DefaultAddressService`.
There are secondary paths: the CIP-30 wallet API uses Yaci Store's Blockfrost-compatible API for wallet queries/build/submit flows, and Yano-only mode uses Yano's Blockfrost-compatible HTTP API. For YACD, this reinforces a cautious stance: avoid inventing a broad wallet layer early, but a narrow faucet/topup helper may still be useful if existing tools cannot easily submit funded transactions against the local node.

## 2026-05-20 13:31 — Existing faucet service scan
Current faucet scan found one serious upstream candidate: `input-output-hk/cardano-faucet`. It is current enough to have a May 2026 `11.0` release for `cardano-node 11.0.1`, uses `cardano-api`, reads `CONFIG_FILE` and `CARDANO_NODE_SOCKET_PATH`, connects to the local node socket, exposes `/send-money`, `/delegate`, metrics/basic UI endpoints, supports API-key and reCAPTCHA limits, derives its funding key from a mnemonic, tracks faucet UTxOs, signs transactions, and submits them through local tx submission.
Fit concern: IOG's faucet is built for public preview/preprod-style faucet operation, not arbitrary local devnet topups. It appears to distribute fixed configured bucket sizes by consuming matching pre-split UTxOs, has no release assets, and would require us to package/configure it for Kubernetes. Other GitHub candidates (`spectrum-finance/cardano-faucet`, `Snowstorm08/Cardano-Faucet-Service`) are older/small forks with Dockerfiles that expect prebuilt binaries and are not strong dependencies. Cardano Wallet is a possible existing wallet-holding service, but it is heavier and not a faucet. Current lean recommendation remains: do not commit to a custom service in the MVP; first see whether the CLI can perform topup using mounted/generated secrets and existing node/Ogmios/Yaci Store APIs. If an in-cluster service is needed, consider IOG faucet as a reference or package candidate, but expect adaptation.

## 2026-05-20 13:36 — Blockfrost vs Ogmios for submit/topup
API comparison checkpoint: both Blockfrost-compatible APIs and Ogmios can submit already-built/signed transaction CBOR, and neither is a wallet or faucet by itself. Ogmios is a direct JSON-RPC/WebSocket bridge over node mini-protocols; it exposes `submitTransaction` and `evaluateTransaction`, but explicitly expects callers to know how to produce and sign serialized transactions. Blockfrost is an HTTP/REST application API and index/query surface; hosted Blockfrost and compatible local services expose convenient SDK-oriented query endpoints and transaction submission, while local Blockfrost Platform can expose `/tx/submit` without a data node and more query endpoints when paired with Dolos.
Design implication: for YACD's first core service set, Ogmios is the better low-level companion to a node because it supports direct local transaction submission/evaluation without an indexer. A Blockfrost-compatible service is more valuable as an optional SDK/indexer profile for app developers, not as the minimum faucet/topup substrate. A bespoke topup helper could use Ogmios or direct node mini-protocols, and only needs Blockfrost/Yaci Store if we choose to rely on indexed address/UTxO queries.

## 2026-05-20 13:48 — Chain access contract for custom services
Clarified design question: custom YACD services are not replacing Ogmios or Blockfrost; they need a preferred way to talk to the chain. Current recommendation is to treat Ogmios as the first default chain-access dependency for YACD-owned services because it is close to the node, does not require an indexer, and covers submit/evaluate plus node-query style interactions. A custom faucet can hold YACD-managed funding keys, build/sign transactions itself, then submit through Ogmios or direct local tx submission.
Blockfrost-compatible APIs should remain optional indexed/app-facing services. They are useful when a service or user workflow needs REST/SDK compatibility, address/transaction lookup, or historical/indexed data, but they should not be required for the first faucet/topup path. Keep the implementation behind a small internal chain client interface so YACD can start with an Ogmios adapter and add a Blockfrost/Yaci Store adapter only if indexed queries become necessary.

## 2026-05-20 13:52 — Yaci Store vs Blockfrost deployment shape
Deployment comparison checkpoint: Yaci Store is essentially a Spring Boot indexer/API app plus durable database. Its Docker distribution runs `bloxbean/yaci-store` with PostgreSQL, mounts config/logs/data/plugins, exposes API/Swagger on 8080, and uses Actuator health/prometheus endpoints. It indexes through node-to-node settings (`store.cardano.host`, `port`, `protocol-magic`), optionally uses node-to-client socket/TCP for protocol parameters and transaction submission, can use Submit API or Ogmios for transaction submission/evaluation, and needs custom genesis files for custom networks. The operational surface includes database sizing, store/API flags, parallel processing, sync start/stop points, post-sync index application, rollback/admin CLI, and optional admin UI/recovery behavior.
Blockfrost Platform is newer and split differently. The platform process can run in solitary mode with a Cardano node socket and expose health/metrics/genesis/transaction submit without a data node. Full Blockfrost-style query endpoints require Dolos as a separate data node, with its own storage volume, generated TOML config, network/genesis/upstream/Mithril settings, and service endpoint wired into `blockfrost-platform --data-node`. Its Docker Compose flow includes node initialization, cardano-node, Dolos, and Blockfrost Platform.
Current read: for transaction submission only, Blockfrost Platform is simpler than Yaci Store, but we likely do not need either if Ogmios is already in the core service set. For indexed Blockfrost-compatible app APIs, Yaci Store is heavier internally but more self-contained from a controller perspective: one app plus DB. Blockfrost Platform becomes a multi-component platform + Dolos + node-socket lifecycle, which may be harder for a first YACD controller to maintain. Neither should be mandatory in the MVP; if we add an optional Blockfrost-compatible profile early, Yaci Store may be the more direct first CRD spike despite its Java/Spring footprint.

## 2026-05-20 13:56 — Node socket access model
Socket design checkpoint: do not treat the Cardano node Unix socket as a normal cluster endpoint. A Unix domain socket is a pod/node-local filesystem IPC object, so the clean Kubernetes-native way to share it is within one Pod via a shared `emptyDir` mounted at a stable path such as `/ipc/node.socket`. The node writes the socket; socket-native companion containers such as Ogmios can consume it as sidecars. The node data directory should stay on its own PVC; the socket directory should be ephemeral.
For supporting services outside the node Pod, prefer network APIs instead of raw socket access. The first service path should be node + Ogmios sidecar, then custom services and the CLI talk to Ogmios over a ClusterIP. Services that can index through node-to-node TCP, such as Yaci Store, can run in separate Pods and use a Service for that connection. Services needing node-to-client access can either use Ogmios/Submit API, support n2c over TCP, or require a deliberate proxy pattern.
Avoid making RWX PVC or hostPath socket sharing part of the design. It is fragile, scheduler-sensitive, and a poor fit for hosted clusters. If a future service absolutely requires a Unix socket but cannot be colocated, the safer experimental pattern is a node-side n2c TCP proxy plus an optional consumer-side proxy that recreates a local Unix socket; that should remain an advanced/verified profile, not MVP behavior. Design implication: socket-requiring services are "node-local attachments" and may need special treatment in the API/controller model rather than being fully independent Deployments.

## 2026-05-20 14:05 — CRD composition pressure from IPC services
CRD implication checkpoint: separate service CRDs do not have to imply separate Pods. If a service requires raw node IPC access, the runtime topology may still be a sidecar in the node Pod. Kubernetes cannot add a sidecar to an already-running Pod; changing the sidecar set means changing the StatefulSet Pod template and recreating the node Pod. That restart is probably acceptable for local-first dev environments, but it must be visible in status and should not be a surprising hidden side effect.
Avoid multiple controllers independently patching the same node StatefulSet. The clean ownership rule is that one controller owns the rendered node Pod template. Supporting service CRDs can reference the environment and express desired capabilities, but the environment/node controller should watch those CRDs, aggregate accepted node-local attachments, and render the final StatefulSet. Service controllers can own separate resources such as Services, ConfigMaps, databases, or status, but not the node Pod template.
Near-term MVP recommendation: keep Ogmios as an inline/default node-local attachment in the primary environment CRD rather than designing a generic attachment framework immediately. Treat later socket-bound services like db-sync as either inline environment features or service CRDs that the primary controller composes into the node Pod. Network-only services can remain independently reconciled Deployments. If node restarts become too disruptive, evaluate a deliberately managed n2c proxy profile, but do not start there.

## 2026-05-20 14:08 — Dedicated follower nodes for socket-bound services
Alternative socket model: for a multi-node YACD network, heavyweight socket-bound services can get their own follower/relay node in the same Pod. The service consumes a local Unix socket from that follower node, while the follower connects to the primary/bootstrap node over normal node-to-node TCP. This avoids mutating or restarting the main node Pod when adding a service such as db-sync, and lets the service CRD own its own StatefulSet, PVCs, database, service config, and status.
Tradeoff: this spends more CPU, memory, storage, and startup time because each attached service may sync or follow through its own node process. That is probably acceptable for opt-in indexers and team/shared clusters, but too expensive as an automatic default for every helper. The CRD surface can make this an explicit access mode, e.g. network API, primary-node sidecar, or dedicated follower node.
Current direction: keep one primary node for the smallest MVP, with Ogmios as a sidecar/default API. For db-sync and similar indexers, prefer the dedicated follower-node pattern as the first serious service-CRD design because it preserves independent reconciliation and avoids primary node restarts. The primary environment CRD should expose enough network/bootstrap status for service controllers to configure follower nodes safely.

## 2026-05-20 14:17 — Provisional architecture alignment
Provisional aligned shape: YACD should have a primary environment/network CRD that owns the core Cardano node substrate and provides the tuning surface for local test scenarios. The first implementation can be one primary node with Ogmios as a default sidecar, but the API should not be named so narrowly that it cannot later describe multi-node topology, bootstrap peers, genesis material, or service attachment data.
Supporting services should be separate CRDs with separate controllers. They reference the primary environment to derive chain/network information. Network-only services reconcile independent Deployments. Heavy IPC services such as db-sync should prefer the dedicated follower-node pattern so the service controller owns its own node+service Pod, PVCs, database wiring, and status without restarting the primary node. Yaci Store remains a candidate optional Blockfrost-like/indexer profile, but db-sync is the better first supporting-service priority because it directly matches the team's existing internal workflow.
The product surface remains two-tiered: a developer-facing config checked into a repo and compiled by a companion CLI into Kubernetes CRDs, plus CLI subcommands for imperative operations such as topups. A bespoke faucet/topup service should stay narrow and use Ogmios for chain interaction; wallet generation can be declarative/bootstrap-oriented, while ad hoc topups stay imperative.
